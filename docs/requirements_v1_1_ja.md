# Amazon Product Discovery Engine 要件定義書 v1.1

> **対象**: 自分1人で使う個人ツール
> **ゴール**: Amazon FBA / OEM 候補商品を「迷わず捨て、迷わず GO する」状態に持ち込む
> **更新日**: 2026-05-08

---

## 1. 目的とスコープ

### 1.1 目的

Amazon 物販の「カテゴリ起点リサーチ → 仕入れ判断」を、**経験則ではなく構造ルール**で再現可能にする個人ツール。

価値の核は3つ:

1. **発掘**: カテゴリから候補 ASIN を網羅的に集める
2. **判定**: 市場性 + 利益性 + 実行可能性を統合スコアと**強制ゲート**で決める
3. **学習**: 自分の判断履歴と販売結果を蓄積し、ツールが育つ

### 1.2 ゴール / アンチゴール

#### ゴール

- 1カテゴリの探索 → 詳細判定までを **15分以内**で完了する
- 1 ASIN あたりの判断時間: **1分以下**
- GO 判定の事後妥当性 (実際に仕入れて損をしなかった率): **70%以上**
- 同一商品を1ヶ月後に再判定したときの判定一致率: **90%以上**

#### アンチゴール (やらないこと)

- 多人数同時利用 / チーム機能 (ロール、共有、コメント)
- リアルタイム在庫追跡 / 大規模スクレイピング
- Amazon 以外マーケットプレイスの判定
- 商品自動仕入れ / 自動発注
- 公開 API / SaaS 化 / マルチテナント

### 1.3 利用者

- 個人 (運用者は1名)
- Amazon Seller Central を運用している前提
- 物販判断のドメイン知識は持つが、エンジニアではない

## 2. プロダクト原則

1. **データを見せるな、決断させろ** — 数値羅列ではなく GO / 条件付き GO / NO-GO を必ず出す
2. **利益が残るかまで判定** — 「売れそう」では不十分。FBA 手数料・広告費・粗利まで含めて判定
3. **構造ルール優先** — 経験則ではなく、明示的な閾値とゲートで判定する
4. **強い NO-GO > 弱い GO** — 迷ったら捨てる。GO には根拠が必要
5. **再現性 > 直感** — 同じ入力に同じ出力。判断根拠は常に説明可能
6. **学習する** — 過去判断と販売結果を蓄積、ツールが個人の経験を取り込む
7. **個人ツールに最適化** — 多人数前提の機能は持ち込まない

## 4. 機能要件

### 4.1 商品探索

#### 入力

| 項目 | 必須 | デフォルト | 備考 |
|---|---|---|---|
| カテゴリ | ✓ | — | フリーテキスト or プリセット |
| 価格帯 | | `¥3,000〜¥8,000` | 推奨ゾーン |
| レビュー数上限 | | `500` | 競合参入余地の閾値 |
| 候補件数上限 | | `50` | 上限 100 |
| 除外辞書適用 | | ON | UC-07 の辞書を反映 |

#### 処理

1. LLM で探索キーワードを **8個以上**自動生成
2. 各キーワードで Amazon を検索 (複数ページ)
3. 候補 ASIN を統合し重複除去
4. Keepa API で時系列データ取得 (価格 / BSR / 出品者)
5. 必要に応じて Amazon SP-API で手数料・カタログ補完
6. 初期スコアリング → ゲート判定 → ソート

#### キーワード生成要件

最低 8 キーワード、以下の軸を**必ず含む**:

- 用途軸 (例: デスク周り → "デスク 整理")
- 問題解決軸 (例: "ケーブル ごちゃごちゃ 解消")
- サイズ / 形状軸 (例: "コンパクト デスクトレイ")
- セット / 付属品軸 (例: "デスクマット セット")
- 利用シーン軸 (例: "在宅ワーク デスク")

#### 探索段階の自動除外

以下は初期探索段階で除外し、`discovery_runs.excluded_candidates` に**理由付き**で記録:

| 除外理由 | 検出方法 |
|---|---|
| 大型商品 | サイズ区分 LARGE 以上 |
| 重量過多 | 1,000g 超 |
| 規制カテゴリ | 規制カテゴリリストとの一致 |
| 危険物 | 危険物フラグ (SP-API or 辞書) |
| 強ブランド独占 | 上位3ブランドで売上シェア > 70% |
| 個人除外辞書 | UC-07 で登録した除外条件 |

#### 出力

- 候補商品リスト (デフォルトでスコア降順)
- 各候補の初期スコア + 簡易判定
- 除外候補は別ペインで「除外理由」とともに表示 (透明性確保)

### 4.2 スコアリング (構造ルールベース)

#### 5軸スコア (合計 100)

| 軸 | 配点 | 趣旨 |
|---|---|---|
| 価格適正 | 25 | 個人物販で扱える価格帯か |
| サイズ効率 | 20 | FBA 手数料が利益を圧迫しないか |
| 競争余地 | 20 | 後発で勝負になるか |
| 価格安定性 | 15 | 価格崩壊リスクは低いか |
| OEM 適性 | 20 | 差別化と再現性は確保できるか |

#### 価格適正 (0-25)

| 区分 | 価格帯 | スコア |
|---|---|---|
| 推奨 | ¥3,000〜¥8,000 | 25 |
| 注意 (低) | ¥1,500〜¥2,999 | 15 |
| 注意 (高) | ¥8,001〜¥12,000 | 15 |
| 非推奨 | ¥1,500未満 | 0 |
| 非推奨 | ¥12,001以上 | 5 |

#### サイズ効率 (0-20)

| 区分 | 重量 / サイズ | スコア |
|---|---|---|
| 推奨 | 500g 以下 かつ 標準小型 | 20 |
| 注意 | 501〜1,000g かつ 標準 | 12 |
| 非推奨 | 1,000g 超 or 大型区分 | 0 |

#### 競争余地 (0-20)

| 区分 | レビュー数 + ブランド支配 | スコア |
|---|---|---|
| 推奨 | レビュー 100〜500 かつ 強ブランド支配なし | 20 |
| 注意 | レビュー 501〜1,000 | 10 |
| 非推奨 | レビュー 1,000 超 or 強ブランド支配 | 0 |

#### 価格安定性 (0-15)

Keepa から計算する派生指標 `price_stability_index` で評価:

| 過去90日の価格変動係数 (CV) | スコア |
|---|---|
| ≤ 0.10 | 15 |
| 0.10〜0.20 | 10 |
| 0.20〜0.30 | 5 |
| > 0.30 | 0 (価格崩壊リスク) |

セール頻度: 過去90日でセール価格期間が**全体の 30% を超える**場合は **-3点**

#### OEM 適性 (0-20)

以下4観点をそれぞれ 0-5 で評価し合算 (LLM + ルールハイブリッド):

| 観点 | 0 (低) | 5 (高) |
|---|---|---|
| 工場再現性 | 高度な金型 / 特殊素材必須 | 既製 OEM で対応可能 |
| 差別化余地 | 完成度高くレビュー満足度 4.5+ | 不満レビュー多数 / 改善余地明確 |
| 技術複雑性 | 電子部品 / 認証必須 | 単純構造 |
| ブランド非依存度 | 上位の 8割が単一ブランド | 多ブランド分散 |

### 4.3 ゲート判定 (強制 NO-GO 条件)

スコアと**独立**に判定する。ヒットすれば判定は最大「条件付き GO」または「NO-GO」に強制ダウングレード。

| ゲート | 条件 | 効果 |
|---|---|---|
| 粗利率不足 | 粗利率 < 30% | NO-GO |
| 広告耐性不足 | 想定広告費 > 粗利の 50% | NO-GO |
| 月販不足 | 想定月販 < 100個 | 条件付き GO 上限 |
| 価格崩壊 | 過去90日の価格下落率 > 20% | NO-GO |
| 規制 / 危険物 | 規制対象カテゴリ or 危険物 | NO-GO |
| 知財リスク | 特許 / 商標で訴訟事例 or 出願ヒット | NO-GO |
| OEM 再現困難 | OEM 適性スコア ≤ 5 | 条件付き GO 上限 |
| ブランド支配 | 上位3ブランドで売上シェア > 70% | 条件付き GO 上限 |

#### 判定ロジック

```
合計スコア >= 75 かつ ゲート全クリア             → GO
合計スコア >= 60 かつ NO-GO ゲートなし          → 条件付き GO
それ以外                                          → NO-GO
```

---

### 4.4 利益性計算

#### 必須項目

| 項目 | 算出方法 |
|---|---|
| 想定販売価格 | Keepa median 90d |
| Amazon 販売手数料 | カテゴリ別手数料率 × 販売価格 |
| FBA 手数料 | SP-API Product Fees or 辞書ベース推定 |
| 想定原価 | 販売価格 × 原価率 (デフォルト 30%) |
| 想定広告費 | CPC ÷ CVR = 1成約あたり広告費 |
| 粗利 | 販売価格 − 全コスト |
| 粗利率 | 粗利 ÷ 販売価格 |

#### 広告費試算

```
想定 CPC: ¥50〜¥150 (中央値 ¥100)
想定 CVR: 10% (ユーザ調整可能)
1成約あたり広告費 = CPC ÷ CVR = ¥1,000 (中央値時)
```

これが**粗利の 50% 以下**でない場合、広告耐性不足ゲート発動

#### 表示

- 全項目を商品詳細画面に内訳として表示
- 推定パラメータ (原価率、CVR、CPC) は調整可能、保存可能

### 4.7 LLM 分析機能

#### 使用モデル

- **第一**: Google Gemini (有料枠、商用利用条件を遵守)
- **代替候補**: OpenAI / Anthropic API
- Provider Adapter パターンで切替可能にする

- 低評価レビューの示唆抽出 (Amazon から取得した上位低評価レビューを入力)

#### プロンプト管理

- プロンプトテンプレートはリポジトリ内でバージョン管理
- 分析レポートには `prompt_version` を記録 (再現性)

### 4.8 監視リスト / 再評価 (UC-05)

#### 監視リスト

- GO / 条件付き GO 判定 + 「監視リスト追加」した ASIN
- 状態: `candidate` (検討中) / `sourcing` (仕入れ進行中) / `live` (販売中)

#### 再評価 (Cron 日次)

1. 監視リスト取得
2. Keepa を再取得
3. スコア + ゲートを再計算
4. 前回との差分検出
5. 重大変化を**通知**
6. `analysis` に新エントリで保存 (履歴保持・上書きしない)

#### 通知

- メール または Webhook (Slack / Discord)
- ペイロードに変化サマリと商品リンク

#### 辞書の種類

| 辞書 | 内容 | 用途 |

### 4.10 判断履歴 (UC-06)

- 全 ASIN 判定を時系列で記録 (`analysis` テーブル、上書きしない)
- 同一 ASIN の再評価は履歴として保持
- 仕入れ後フィードバック (実際に仕入れたか / 黒字か赤字か / メモ) を記録可能
- 後追い集計画面で「過去 GO 判定の事後妥当性」を表示 (Phase 4)

---

### 4.11 キャッシュ

#### 階層化

| 層 | TTL | 内容 |
|---|---|---|
| Keepa 生データ | 24時間 | 価格 / BSR / 出品者推移 |
| Keepa 派生指標 | 24時間 | CV、トレンド指標など |
| LLM 分析レポート | 7日間 | 短期で内容が変わらない |
| ゲート判定 / スコア | 24時間 | 入力データに連動 |

#### 強制リフレッシュ

- 商品詳細画面に「最新を取得」ボタン
- 押下時に API コスト消費を確認ダイアログで明示

---

### 4.12 コスト可視化

#### 表示

- ダッシュボードに「今月の API 利用コスト」(Keepa / LLM / SP-API) を表示
- 月予算 (デフォルト ¥10,000) を超えた場合は警告
- API 呼び出し履歴を最低 30日保持

#### 制御

- 月予算の **80% 超過** → 「キャッシュ優先モード」自動切替 (新規 API 呼び出しを抑制)
- 月予算の **100% 超過** → API 呼び出し停止 (キャッシュのみ利用)
- 強制リフレッシュは制御を上書き可能 (確認ダイアログ付き)

## 5. 非機能要件

### 5.1 パフォーマンス

| 操作 | 目標 |
|---|---|
| 一覧表示 (キャッシュヒット時) | 1秒以内 |
| 一覧表示 (新規探索) | 30秒以内 (LLM + API 呼び出し含む) |
| 詳細分析 (キャッシュヒット時) | 1秒以内 |
| 詳細分析 (新規) | 5秒以内 |

### 5.2 可用性 / 失敗時の挙動

- 個人利用前提、SLA は設定しない
- 外部 API 失敗時は **指数バックオフ最大3回**リトライ
- 取得不可データがあっても**部分結果**を返す + UI で「一部取得失敗」を明示
- 失敗時のフォールバック:
  - **LLM 失敗**: スコア + ゲート判定はルールのみで成立。レポート部分のみ「取得失敗」表示
  - **Keepa 失敗**: キャッシュデータで暫定判定 + 「最新データ未取得」明示
  - **SP-API 失敗**: 辞書ベースの手数料推定にフォールバック

### 5.3 セキュリティ

- 認証: Supabase Auth (単一ユーザ運用)
- API キーはサーバ側環境変数で管理 (`KEEPA_API_KEY` / `GEMINI_API_KEY` / `SP_API_*`)
- Cron エンドポイントは `CRON_SECRET` で保護
- 個人情報は取り扱わない (商品情報のみ)

### 5.4 コスト

- 月額 **¥10,000 以内**
- 内訳目安:
  - Vercel: ¥0〜3,000
  - Supabase: ¥0〜3,000
  - Keepa API: ¥1,000〜3,000
  - LLM (Gemini): ¥500〜2,000
- 4.12 のコスト可視化で月次管理

### 5.5 データ管理

- バックアップ: Supabase 自動バックアップに依存
- データエクスポート: ASIN 一覧 + 判定結果を CSV で書き出せる
- データ移行: SQL ダンプで自前移行可能なスキーマ設計

### 5.6 開発 / 運用

- ローカル開発: Next.js dev server + Supabase ローカルエミュレータ
- デプロイ: Vercel (main ブランチ push で本番デプロイ)
- 環境分離: development / production (Supabase プロジェクト分離)

## 7. 外部 API 方針

### 7.1 Keepa API

主用途:

- 価格推移 / BSR 推移 / 出品者数推移
- 価格崩壊検知 (`price_stability_index`)
- セール頻度検知

### 7.2 Amazon SP-API

MVP では必須としない。Phase 3 で導入し利益判定精度を上げる。

利用候補:

- Product Fees API (FBA 手数料正確化)
- Catalog Items API (サイズ / 重量 / 危険物フラグ補完)
- Product Pricing API (オファー価格補完)

### 7.3 採用しない API

- Product Advertising API (PA-API): レート制限と申請ハードルから新規採用しない

### 7.4 抽象化方針

- LLM は **Provider Adapter** で Gemini / OpenAI / Anthropic を切替可能
- Keepa は単一プロバイダだがデータアクセス層を分離 (将来の SP-API 統合に備える)

## 9. 処理フロー

### 9.1 探索フロー

```
カテゴリ入力
  → キーワード生成 (LLM, 8個以上)
  → 各キーワードで Amazon 候補取得
  → ASIN 統合 + 重複除去
  → 学習辞書フィルタ (除外ブランド / NG パターン)
  → Keepa 一括取得
  → 初期スコアリング + ゲート判定
  → ソート → 一覧表示
  → discovery_runs に記録
```

### 9.2 詳細分析フロー

```
ASIN 選択
  → DB キャッシュ確認
  → (ヒット & 有効) 即時表示
  → (ミス or 期限切れ)
      → Keepa 取得
      → SP-API 補完 (Phase 3+)
      → 5軸スコアリング
      → ゲート判定
      → 利益性計算
      → LLM 分析
      → analysis に保存
  → 表示
```

### 9.3 監視 / 再評価フロー (Cron 日次)

```
監視リスト取得
  → Keepa 再取得
  → スコア + ゲート再計算
  → 前回との差分検出
  → 重大変化を通知
  → analysis に新エントリで保存 (履歴保持)
```

## 付録 A: 用語集
